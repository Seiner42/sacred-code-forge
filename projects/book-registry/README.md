# book-registry

Реестр прочитанных книг.

Этот проект — каркас под будущий механизм учёта прочитанного: хранение записей о книгах, статуса чтения, заметок и, при необходимости, связанной автоматизации.

## Назначение

- вести список прочитанных книг;
- хранить базовые метаданные о книгах;
- отмечать даты, статус и личные заметки;
- стать основой для будущего сервиса, CLI или иной формы интерфейса.

## MVP

Первая версия проекта предназначена для простого хранения списка уже прочитанных книг.

### Что хранит запись о книге

Каждая запись должна содержать:

- `id` — уникальный идентификатор записи;
- `title` — название книги;
- `author` — автор;
- `rating` — личная оценка по шкале от `1` до `10`;
- `finishedAt` — дата прочтения в формате `YYYY-MM-DD`;
- `comment` — личный комментарий к книге.

### Формат хранения

Для MVP данные хранятся в одном JSON-файле.

Рекомендуемый путь:
- `data/books.json`

Для этого проекта `data/books.json` считается рабочим файлом данных и может храниться в git, так как записи о книгах не являются секретом и должны легко восстанавливаться на другой машине.

Базовая структура файла:

```json
{
  "books": []
}
```

### Цель MVP

MVP должен позволять:

1. хранить записи о прочитанных книгах;
2. добавлять новые записи;
3. просматривать существующий список;
4. сохранять данные в простом и переносимом формате без отдельной базы данных;
5. защищать доступ к приложению простым паролем и cookie-сессией.

### Архитектура MVP

Для первой версии предлагается лёгкая веб-архитектура:

- **frontend** — одна простая веб-страница на HTML/CSS/JavaScript без тяжёлого фреймворка;
- **backend** — лёгкий HTTP-сервер на Node.js;
- **storage** — JSON-файл `data/books.json`.

В MVP frontend и backend работают как **одно приложение**:

- один Node.js-сервис;
- один процесс;
- один порт;
- frontend раздаётся этим же сервером как статические файлы;
- API обслуживается тем же приложением.

Такой подход выбран, чтобы MVP был:

- простым в разработке;
- быстрым в запуске;
- понятным в сопровождении;
- пригодным для последующего расширения.

### Минимальная аутентификация

Для MVP доступ к приложению должен быть закрыт одним паролем.

Принцип работы:

1. пользователь открывает веб-страницу;
2. frontend показывает форму ввода пароля;
3. пароль отправляется на backend;
4. backend проверяет пароль;
5. при успехе backend устанавливает cookie-сессию;
6. дальнейший доступ к API разрешён только при наличии валидной cookie.

Требования:

- пароль хранится только на backend, например в переменной окружения;
- frontend не содержит правильного пароля и не выполняет проверку как источник истины;
- cookie должна быть `HttpOnly`;
- рекомендуется `SameSite=Strict`;
- `Secure` следует включать только при реальном HTTPS;
- для этого в конфигурации используется отдельный флаг `COOKIE_SECURE=true|false`.

### Минимальный API

Для MVP достаточно такого набора эндпоинтов:

- `POST /api/login` — проверка пароля и установка cookie-сессии;
- `POST /api/logout` — завершение сессии;
- `GET /api/books` — получение списка книг;
- `POST /api/books` — добавление новой записи.

### Почему выбран JSON

JSON выбран для первой версии, потому что он:

- прост в чтении и ручном редактировании;
- удобен для git;
- не требует отдельной инфраструктуры;
- позволяет позже без лишней боли перейти на SQLite или другой формат хранения.

## Деплой MVP

Предпочтительный вариант деплоя для первой версии:

- одно Node.js-приложение;
- один сервис;
- при контейнеризации — **один Docker-контейнер**;
- один внешний порт;
- отдельный volume или bind mount для `data/books.json`, чтобы данные не терялись при пересоздании контейнера.

Это означает, что контейнер одновременно:

- раздаёт frontend;
- обслуживает backend API;
- работает с файловым хранилищем книги.

## Локальный запуск

```bash
cp .env.example .env
npm install
npm start
```

По умолчанию приложение будет доступно на `http://localhost:3000`.

## Docker-запуск

Перед запуском контейнера следует подготовить `.env` и файл данных:

```bash
cp .env.example .env
cp data/books.example.json data/books.json
docker compose up --build -d
```

В `docker-compose.yml` файл `data/books.json` подключён в контейнер как внешний bind mount, чтобы записи сохранялись вне жизненного цикла контейнера.

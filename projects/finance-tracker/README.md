# Finance Tracker

Личный финансовый трекер на **Next.js + SQLite**.

Проект заточен под:
- ручной учёт операций;
- импорт **Альфа CSV**;
- нормализацию продавцов через **merchant rules**;
- отдельный контур подписок;
- простую локальную админку для категорий и правил;
- запуск в Docker.

---

## Что умеет приложение

- показывает сводку по текущему месяцу;
- хранит и редактирует операции вручную;
- импортирует выписки **Альфа Банка**;
- отправляет неоднозначные паттерны в ручной разбор;
- сохраняет и применяет **merchant rules**;
- ведёт список подписок;
- даёт редактировать категории и merchant rules через `/admin`;
- умеет массово переиндексировать уже сохранённые операции по текущим правилам;
- автоматически поднимает **дефолтные категории и merchant rules** на чистой базе.

---

## Разделы

### Обзор (`/`)

Главная страница показывает:
- **расходы за месяц**;
- **пополнения**;
- **количество операций**;
- **подписки в этом месяце**;
- последние операции месяца;
- топ категорий месяца;
- краткий блок по подпискам.

Особенности:
- расходы подсвечены красным;
- пополнения — зелёным;
- подписки месяца — циановым акцентом;
- в топе категорий и в последних операциях используются иконки категорий.

---

### Операции (`/operations`)

Экран операций поддерживает:
- список всех операций из SQLite;
- фильтрацию:
  - по месяцу;
  - по произвольному диапазону дат;
  - по типу (`расход` / `пополнение`);
  - по категории;
  - по поиску (название или комментарий);
- создание, редактирование и удаление операций;
- смену категории при редактировании операции;
- отдельный блок сводки по текущей выборке:
  - количество операций;
  - сумма приходов;
  - сумма расходов.

Важно:
- отдельного смыслового типа **`transfer`** в UI больше нет;
- положительные движения считаются **пополнениями**;
- отрицательные — **расходами**.

---

### Подписки (`/subscriptions`)

Экран подписок поддерживает:
- ручной список регулярных списаний;
- месячные и годовые подписки;
- создание, редактирование и удаление;
- две итоговые карточки:
  - **всего в месяц** — только ежемесячные подписки;
  - **всего в год** — месячные × 12 + годовые;
- человекочитаемый статус оплаты на UI:
  - **Оплачено** — зелёным;
  - **Не оплачено** — красным.

Логика определяется по текущему циклу:
- если дата списания в текущем цикле уже прошла — подписка считается **оплаченной**;
- если ещё не прошла — **не оплаченной**.

На главной карточка **«Подписки в этом месяце»** считает:
- все ежемесячные подписки;
- годовые — только если сейчас месяц их оплаты.

---

### Импорт (`/imports`)

Импорт принимает только **Альфа CSV**.

Что делает импортёр:
- читает CSV через `csv-parse`;
- нормализует даты, суммы, валюту и продавцов;
- определяет направление операции как:
  - `income` для пополнений;
  - `expense` для списаний;
- применяет:
  - встроенные дефолтные merchant rules;
  - пользовательские merchant rules из БД;
- защищает импорт от дублей через `source_row_hash`;
- создаёт review items для сомнительных продавцов;
- сохраняет историю импортов.

#### Статистика импорта

Для каждого файла сохраняется:
- сколько строк было в CSV;
- сколько импортировано;
- сколько пропущено как дубли;
- сколько ушло в разбор;
- сколько завершилось ошибкой.

---

### Разбор (`/review`)

Экран разбора нужен для паттернов, которые система не смогла уверенно нормализовать.

Вверху страницы есть общий блок **«Что сделать»**, а дальше идут карточки по конкретным продавцам.

На карточке можно:
- подтвердить, как назвать продавца;
- выбрать категорию;
- выбрать тип совпадения правила:
  - `contains`;
  - `exact`;
- сохранить правило;
- или пропустить элемент.

Дополнительно есть кнопка:
- **«Обновить разбор»** — заново прогоняет pending-элементы по текущим merchant rules и автоматически убирает те, которые уже стали распознаваться.

После сохранения правила:
- создаётся запись в `merchant_rules`;
- уже импортированные операции с этим `raw_merchant` внутри импорта обновляются.

---

### Админка (`/admin`)

Админка содержит два блока:

#### 1. Категории
Можно редактировать:
- название;
- slug;
- цвет;
- иконку;
- активность.

#### 2. Merchant rules
Можно:
- просматривать правила;
- создавать новые;
- редактировать существующие.

У правила есть поля:
- тип совпадения (`exact` / `contains`);
- паттерн;
- нормализованное имя продавца;
- категория;
- override направления (`expense` / `income` / не менять);
- приоритет;
- заметка.

Также в этом блоке есть кнопка:
- **«Переиндексировать операции»** — массово заново применяет текущие merchant rules ко всем уже сохранённым операциям и обновляет:
  - `merchant_normalized`
  - `category_id`
  - `direction` (если правило задаёт override).

## Принципы текущей модели

### 1. Только `expense` и `income`

В проекте больше нет отдельного рабочего типа `transfer`.

Логика такая:
- пополнение → `income`
- списание → `expense`

Даже если исходно в банковской логике это был перевод между своими счетами, внутри трекера это теперь просто положительное или отрицательное движение.

### 2. Все операции участвуют в сводке

Legacy-контур `include_in_reports` удалён из рабочей модели.

Текущий принцип:
- все операции хранятся и участвуют в расчётах;
- аналитика строится по `direction`, категории и датам.

## Импорт Альфа CSV: маршрут данных

При загрузке файла:

1. `POST /api/imports/upload`
2. вызывается `importAlfaCsv()`
3. читаются:
   - `categories`
   - `merchant_rules`
4. создаётся запись в `imports`
5. каждая строка CSV проходит нормализацию
6. новые операции пишутся в `transactions`
7. сомнительные паттерны пишутся в `import_review_items`
8. запись в `imports` обновляется итоговой статистикой

## Что важно помнить

- база данных хранится в `data/finance-tracker.db`;
- файл базы не должен попадать в git;
- порядок строк внутри одного банковского дня хранится через `source_order`;
- дубли импортов блокируются через `source_row_hash`;
- чистая база автоматически сидится дефолтными категориями и merchant rules.
